MBack プロジェクト 開発ドキュメント＆ロードマップ
1. プロジェクト概要
MBack は、C#で開発されたエンタープライズ品質の常駐型自動バックアップシステムである。
設定を管理するGUIアプリケーション（MBack.Config）と、バックグラウンドで24時間365日稼働するWindowsサービス（MBack.Service）の2つのコンポーネントで構成されている。個人開発からスタートし、実際の現場運用における数々の課題（ネットワーク瞬断、排他ロック、ランサムウェア被害など）を乗り越え、実戦投入可能な「Gold Release (v1.0)」へと進化した。

2. 開発の経緯と解決した主要な課題
幽霊ファイル（一時ファイル）と排他ロック問題:
FileSystemWatcher によるリアルタイム監視では、Office系ソフトなどが生成する一時ファイル（一瞬で消えるファイル）や、使用中の排他ロックにより「ファイルが見つからない（FileNotFound）」や「アクセス拒否」のエラーが多発した。これを**「遅延実行キュー（Debounce処理）」**を導入することで解決。

上書きインストール時の設定消失問題:
初期設定では appsettings.json を実行ファイルと同階層に保存していたため、アップデート時に設定が初期化されるバグが発生。保存先をWindowsの共通データ領域である C:\ProgramData\MBack に変更し、安全なアップデートを実現した。

メール送信のタイムアウト・セキュリティ問題:
昨今の厳格なメールサーバー（Gmail等）に対応するため、標準のSMTP送信に加えて「TLS 1.2 / 1.3の強制化」「POP before SMTP対応」、さらに「Googleアプリパスワード」を利用したセキュアな通知ロジックを実装した。

除外設定のワイルドカード誤認:
除外設定において *.tmp のようなワイルドカード（アスタリスク）を使用すると正常に除外されない仕様を明確化。アスタリスクを使用せず .tmp や ~（チルダ）と指定するだけで部分一致（Contains）として機能するシンプルかつ確実なルールを確立した。

3. 実装済みのコア機能（MBack v1.0）
リアルタイム監視 ＋ 遅延実行キュー:
ファイルの変更・作成を検知後、一般ファイルは10秒、データベースファイル（.accdb等）は300秒待機してからコピーを実行し、不完全な状態でのバックアップを防ぐ。

深夜の定期フルスキャン（取りこぼし防止）:
毎日深夜1時に監視元とバックアップ先を全件比較し、OSからの通知漏れや昼間のロックで失敗したファイルを自動的にリカバーする。

世代管理とゴミ箱機能:
既存ファイルを上書きする際、最大50世代（.v1 〜 .v50）まで自動でリネームして履歴を保存。ファイルが削除された場合は、バックアップ先の _TRASH_ フォルダへ移動し、誤操作からデータを守る。

ランサムウェア検知（サーキットブレーカー）:
「60秒間に2000件（設定変更可能）」など、人間には不可能な速度でファイルが変更・削除された場合、即座に異常事態とみなし、バックアップ処理（サービス内部のキュー）を緊急停止（ロック）してNASへの汚染を防ぐ。

メールアラート通知機能:
バックアップ失敗時やサーキットブレーカー発動時に、管理者へメールを送信。通常のエラーは「1時間に1回」に制限してスパム化を防ぎ、緊急停止時は即座に送信する。テスト送信機能もUIに完備。

詳細設定・履歴復元UI:
ログの保存日数（デフォルト60日）やランサムウェア検知の閾値をGUIから変更可能。過去のバックアップ履歴をツリー表示し、UI上から直感的に復元できる機能を搭載。

4. 今後の開発ロードマップ（MBack v2.0 に向けたタスク）
【最優先タスク】
[Task 1] ネットワークドライブ（NAS等）の自動マウント＆認証機能

課題: 現在、NASにバックアップする際は、Windowsの「サービス」管理画面からログオンアカウント（.\ユーザー名 など）を手動で設定する必要がある。

解決案: MBackの設定UI（Config）で「バックアップ先のユーザー名・パスワード」を入力・保存できるようにし、サービス実行時にプログラム側から自動的にネットワーク認証（マウント）を行う仕組みを実装する。

[Task 2] サーキットブレーカー発動時のUI可視化

課題: ランサムウェア検知で内部処理が緊急停止しても、Windowsサービス自体は生きているため、設定ツール上では「サービス: 実行中（緑色）」と表示され、状態が矛盾して見える。

解決案: サービス側が「緊急停止状態フラグ」をファイル等に書き出し、Config側がそれを読み取ってボタンを「緊急停止中（赤色）」にするなど、視覚的に異常を知らせるUIを実装する。

【機能拡張タスク】
[Task 3] 古い履歴の「自動Zip圧縮」機能（エコ機能）

最新の5世代程度はそのまま残し、古い世代（6〜50世代目）は自動的にZip圧縮して保存することで、サーバーのディスク容量を節約する。

[Task 4] 事前 / 事後スクリプト（バッチ）の連携機能

バックアップ処理の直前に指定した .bat や .exe を実行可能にする（例：SQL ServerやAccessのデータベースを安全にダンプ出力してからコピーさせる機能）。

[Task 5] クラウドへの二次転送（3-2-1ルールの徹底）

NASへのバックアップ完了後、重要なデータのみを AWS S3、Google Drive、または外部FTPサーバーやNextcloudへ転送し、物理的な災害に対する耐性を高める。

[Task 6] ステータスレポートの定期配信

エラー通知だけでなく、「前日は〇〇件のファイルを正常にバックアップしました」という稼働サマリーメールを毎日/毎週定期送信し、管理者に安心感を提供する。
